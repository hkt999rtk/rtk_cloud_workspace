# RTK Video Cloud STRIDE Matrix

This matrix supports `../threat_models/rtk_video_cloud-stride-threat-model.md`.
Priorities are initial AppSec rankings and are assumption-driven where target
deployment details are not yet confirmed.

## Trust Boundaries

| ID | Boundary | Data crossing | Channel | Key evidence |
| --- | --- | --- | --- | --- |
| TB1 | Internet app/admin clients -> reverse proxy/TLS -> Video Cloud API | Bearer tokens, WebRTC offers, media requests, admin/device commands | HTTPS/WSS | `docs/private-cloud-deployment.md`, `repos/rtk_video_cloud/README.md`, `repos/rtk_cloud_contracts_doc/AUTH.md`, `repos/rtk_cloud_contracts_doc/STREAMING.md` |
| TB2 | Device -> mTLS/token issuance -> device-scoped API/WebSocket/MQTT | Device certificate, `devid`, access/refresh tokens, telemetry, media uploads, SDP answers | mTLS/HTTPS, WSS, MQTT | `repos/rtk_cloud_contracts_doc/AUTH.md`, `repos/rtk_cloud_contracts_doc/PROVISION.md`, `repos/rtk_cloud_contracts_doc/STREAMING.md` |
| TB3 | Account Manager/Admin BFF -> Video Cloud protected APIs | Admin bearer tokens, tenant/device lifecycle commands, activation/deactivation state | HTTPS, BFF proxying, cross-service channel | `docs/account-manager-admin-boundary.md`, `repos/rtk_cloud_contracts_doc/PROVISION.md` |
| TB4 | Video Cloud -> Postgres/object storage/EMQX/TURN registry | Runtime state, media, firmware, broker messages, TURN nodes/secrets | SQL, filesystem/S3, MQTT, HTTP/HMAC | `docs/private-cloud-deployment.md`, `repos/rtk_video_cloud/README.md`, `repos/rtk_cloud_contracts_doc/STREAMING.md` |
| TB5 | Deployment pipeline/operator -> env files/secrets/release artifacts | DSNs, JWT secrets, MQTT credentials, object keys, deploy keys, release bundles | GitHub secrets, host env files, object storage, SSH/systemd | `docs/private-cloud-deployment.md`, `docs/deployment-secrets-governance.md`, `README.md` |

## STRIDE Rows

| ID | Boundary | STRIDE | Threat / abuse path | Likelihood | Impact | Priority | Existing controls | Gaps / review needs | Recommended mitigations | Detection ideas |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| S1 | TB1/TB2 | Spoofing | Attacker reuses or forges a bearer token, legacy camera token, or certificate-header compatibility credential to impersonate a device or app. | Medium | High | High | Route scope matrix and subject binding are documented; mTLS can bind device identity. | Confirm implementation rejects subject mismatches on every compatibility route. | Add route-level tests for `devid` mismatch, legacy aliases, and token/cert conflict cases; disable legacy header flows in production when migration completes. | Alert on repeated subject mismatch, token replay, and compatibility-route auth failures. |
| S2 | TB2/TB4 | Spoofing | MQTT client spoofs a device identity and publishes device events or consumes commands if broker auth/TLS/firewall policy is weak. | Medium | High | High | Production guidance prefers MQTT TLS, auth, logs, and firewall rules. | Confirm EMQX auth is device-scoped and cannot be bypassed by local/default listener exposure. | Enforce per-device broker credentials or mTLS, private listeners by default, and explicit exposure checks in deployment verify scripts. | Monitor impossible device location changes, duplicate online owners, and broker auth failures. |
| T1 | TB3/TB4 | Tampering | Cross-service lifecycle command or activation state is modified, replayed, or duplicated, granting wrong service options or wrong device ownership. | Medium | High | High | Provisioning contract requires operation ids, canonical service options, command/event evidence, retry/dead-letter state. | Need review of idempotency, replay handling, and cross-service auth in code and broker config. | Sign or authenticate cross-service messages, enforce idempotency keys, reject conflicting duplicate operation ids, and audit before/after state. | Alert on duplicate operation ids with conflicting payload, dead-letter spikes, and activation state divergence. |
| T2 | TB4/TB5 | Tampering | Media, snapshots, firmware, or release artifacts are modified in object storage or deployment storage. | Low | High | Medium | Production-like profile requires immutable release artifacts or pinned images and object storage lifecycle policy. | Integrity verification for media/firmware/release artifacts must be confirmed in implementation. | Add checksums/signatures for firmware and release bundles; restrict object write credentials by service and path. | Track object overwrite/delete events, unexpected firmware checksum changes, and release artifact provenance. |
| R1 | TB2/TB3 | Repudiation | Device token issuance, certificate issuance, provisioning, media access, or admin proxy actions lack enough audit evidence to prove actor, subject, and outcome. | Medium | Medium | Medium | Provisioning and factory issuance contracts require evidence and audit records; Admin BFF is non-authoritative. | Confirm audit coverage, retention, redaction, and correlation ids across account/video/admin boundaries. | Standardize audit event fields for actor, subject, tenant, operation id, source boundary, and result. | Monitor audit gaps, missing correlation ids, and privileged actions without upstream source facts. |
| I1 | TB1/TB2 | Information Disclosure | Media download compatibility tokens, subscriber scope, query-string tokens, or logs expose clips/snapshots across devices or tenants. | Medium | High | High | Auth contract says bearer auth is normal for new clients and compatibility remains limited; logger docs forbid token logging. | Confirm legacy `?token=` compatibility is tightly scoped and logs redact query strings. | Prefer Authorization headers, short-lived download grants, strict subject binding, and no token-bearing URLs in logs/referrers. | Alert on cross-device download denial, token-looking query strings in logs, and unusual media download volume. |
| I2 | TB5 | Information Disclosure | Deployment secrets, DSNs, JWT keys, MQTT credentials, object storage keys, deploy keys, or private certificate assets leak through git, logs, issue bodies, artifacts, or evidence packages. | Medium | Critical | Critical | Workspace and deployment docs explicitly forbid committing secrets and define accepted storage patterns. | Need verify scripts and CI checks cover new security artifacts and release/evidence paths. | Run secret scanning on commits and artifacts, keep env files ignored/root-owned, redact evidence, rotate exposed secrets immediately. | Monitor secret-scan findings, public artifact contents, and unexpected use of old secret versions. |
| D1 | TB1/TB2/TB4 | Denial of Service | WebRTC signaling, SDP payloads, media upload/download, or many concurrent sessions exhaust API workers, device resources, Postgres, object storage, or TURN capacity. | Medium | High | High | Streaming contract defines expiry, timeout, session lifecycle, and resource-limit errors. | Rate limits, request size limits, session caps, and per-device concurrency controls need implementation review. | Enforce edge and app rate limits, body-size caps, per-device/session quotas, TURN capacity checks, and timeout cleanup. | Alert on 408/409 spikes, long pending session count, TURN capacity saturation, and storage growth anomalies. |
| D2 | TB4 | Denial of Service | Public or weakly protected metrics, EMQX dashboard, NATS, PostgreSQL, or raw service ports are abused for resource exhaustion or control-plane disruption. | Low | High | Medium | Deployment docs require private network or authenticated access for metrics, EMQX dashboard, Postgres, and NATS. | Confirm firewall, bind addresses, and reverse proxy route allowlists in target deployment. | Add deployment preflight checks for public listeners; fail production deploy if private services bind public interfaces unexpectedly. | Periodically scan exposed ports from outside the VPC and alert on drift. |
| E1 | TB1/TB3 | Elevation of Privilege | Scope confusion lets `app`, `subscriber`, `device`, or legacy `camera` credentials perform admin-only actions or access another device. | Medium | High | High | Route scope matrix documents accepted scopes and subject rules by route family. | Need systematic tests for every route family and compatibility alias. | Centralize authorization checks, make deny-by-default route policies explicit, and add table-driven auth tests for scope and subject combinations. | Alert on authz denials by route/scope and unexpected successful admin-route calls by non-admin scopes. |
| E2 | TB3 | Elevation of Privilege | Admin BFF proxy expands privilege by treating local console state or cache as authoritative for tenant/device operations. | Medium | High | High | Boundary doc says Account Manager and Video Cloud remain authoritative; Admin cache is non-authoritative. | Confirm production-mode BFF always prefers upstream facts and handles stale/unauthorized/partial failures safely. | Enforce upstream authorization for mutating operations, mark cache-only views as non-authoritative, and fail closed on upstream auth errors. | Monitor BFF operations that lack upstream authorization evidence or use stale cache for privileged decisions. |
